http常见问题总结 ​

HTTP基本概念 ​

HTTP是什么 ​

HTTP 是超文本传输协议，也就是HyperText Transfer Protocol。

HTTP 是一个在计算机世界里专门在「两点」之间「传输」文字、图片、音频、视频等「超文本」数据的「约定和规范」。

那「HTTP 是用于从互联网服务器传输超文本到本地浏览器的协议」，这种说法正确吗？

这种说法是不正确的。因为也可以是「服务器< -- >服务器」，所以采用两点之间的描述会更准确。

HTTP常见状态码 ​

TIP

1xx 类状态码属于提示信息，是协议处理中的一种中间状态，实际用到的比较少。

TIP

2xx 类状态码表示服务器成功处理了客户端的请求，也是我们最愿意看到的状态。

  • 「200 OK」是最常见的成功状态码，表示一切正常。如果是非 HEAD 请求，服务器返回的响应头都会有 body 数据。(HEAD 请求可以让你获取到资源的元信息（如内容的长度、类型、最后修改时间等）)
  • 「204 No Content」也是常见的成功状态码，与 200 OK 基本相同，但响应头没有 body 数据。
  • 「206 Partial Content」是应用于 HTTP 分块下载或断点续传，表示响应返回的 body 数据并不是资源的全部，而是其中的一部分，也是服务器处理成功的状态。

TIP

3xx 类状态码表示客户端请求的资源发生了变动，需要客户端用新的 URL 重新发送请求获取资源，也就是重定向。

  • 「301 Moved Permanently」表示永久重定向，说明请求的资源已经不存在了，需改用新的 URL 再次访问。
  • 「302 Found」表示临时重定向，说明请求的资源还在，但暂时需要用另一个 URL 来访问。

301 和 302 都会在响应头里使用字段 Location，指明后续要跳转的 URL，浏览器会自动重定向新的 URL。

  • 「304 Not Modified」不具有跳转的含义，表示资源未修改，重定向已存在的缓冲文件，也称缓存重定向，也就是告诉客户端可以继续使用缓存资源，用于缓存控制。

TIP

4xx 类状态码表示客户端发送的报文有误，服务器无法处理，也就是错误码的含义。

  • 「400 Bad Request」表示客户端请求的报文有错误，但只是个笼统的错误。
  • 「403 Forbidden」表示服务器禁止访问资源，并不是客户端的请求出错。
  • 「404 Not Found」表示请求的资源在服务器上不存在或未找到，所以无法提供给客户端。

TIP

5xx 类状态码表示客户端请求报文正确，但是服务器处理时内部发生了错误，属于服务器端的错误码。

  • 「500 Internal Server Error」与 400 类型，是个笼统通用的错误码，服务器发生了什么错误，我们并不知道。
  • 「501 Not Implemented」表示客户端请求的功能还不支持，类似“即将开业，敬请期待”的意思。
  • 「502 Bad Gateway」通常是服务器作为网关或代理时返回的错误码，表示服务器自身工作正常，访问后端服务器发生了错误。
  • 「503 Service Unavailable」表示服务器当前很忙，暂时无法响应客户端，类似“网络服务正忙，请稍后重试”的意思。

HTTP常见字段 ​

Host

text
Host: www.A.com
1

有了 Host 字段，就可以将请求发往「同一台」服务器上的不同网站。(通过host可以确定ip)

Content-Length

text
Content-Length: 1000
1

服务器在返回数据时，会有 Content-Length 字段，表明本次回应的数据长度。因为HTTP协议是基于TCP协议工作的，而TCP协议存在粘包的问题，HTTP通过设置回车符、换行符作为 HTTP header 的边界，通过 Content-Length 字段作为 HTTP body 的边界，这两个方式都是为了解决“粘包”的问题。

Connection

Connection 字段最常用于客户端要求服务器使用「HTTP 长连接」机制，以便其他请求复用。HTTP 长连接的特点是，只要任意一端没有明确提出断开连接，则保持 TCP 连接状态。

HTTP/1.1 版本的默认连接都是长连接，但为了兼容老版本的 HTTP，需要指定 Connection 首部字段的值为 Keep-Alive。

text
Connection: Keep-Alive
1

Content-Type

Content-Type 字段用于服务器回应时，告诉客户端，本次数据是什么格式。

text
Content-Type: text/html; Charset=utf-8
1

上面的类型表明，发送的是网页，而且编码是UTF-8。客户端请求的时候，可以使用 Accept 字段声明自己可以接受哪些数据格式。

text
Accept: */*
1

Content-Encoding

Content-Encoding 字段说明数据的压缩方法。表示服务器返回的数据使用了什么压缩格式

GET和POST ​

GET和POST的区别 ​

GET的语义是从服务器获取指定的资源。GET 请求的参数位置一般是写在 URL 中，URL 规定只能支持 ASCII，所以 GET 请求的参数只允许 ASCII 字符 ，而且浏览器会对 URL 的长度有限制（但是HTTP协议本身对 URL长度并没有做任何规定）。

POST 的语义是根据请求负荷（报文body）对指定的资源做出处理。POST 请求携带数据的位置一般是写在报文 body 中，body 中的数据可以是任意格式的数据(不一定得是application/json格式，还可以是multipart/form-data、application/xml等等)，只要客户端与服务端协商好即可，而且浏览器不会对 body 大小做限制。

GET和POST的幂等性和安全性比较 ​

  • 在 HTTP 协议里，所谓的「安全」是指请求方法不会「破坏」服务器上的资源。
  • 所谓的「幂等」，意思是多次执行相同的操作，结果都是「相同」的。

从 RFC 规范定义的语义来看：

  • GET 方法就是安全且幂等的，因为它是「只读」操作，无论操作多少次，服务器上的数据都是安全的，且每次的结果都是相同的。
  • POST 因为是「新增或提交数据」的操作，会修改服务器上的资源，所以是不安全的，且多次提交数据就会创建多个资源，所以不是幂等的。

实际过程中，开发者不一定会按照 RFC 规范定义的语义来实现 GET 和 POST 方法。

  • 可以用 GET 方法实现新增或删除数据的请求，这样实现的 GET 方法自然就不是安全和幂等。
  • 可以用 POST 方法实现查询数据的请求，这样实现的 POST 方法自然就是安全和幂等。

GET 请求可以带 body 吗？

RFC 规范并没有规定 GET 请求不能带 body 的。理论上，任何请求都可以带 body 的。只是因为 RFC 规范定义的 GET 请求是获取资源，所以根据这个语义不需要用到 body。

另外，URL 中的查询参数也不是 GET 所独有的，POST 请求的 URL 中也可以有参数的。

HTTP缓存技术 ​

对于一些具有重复性的 HTTP 请求，比如每次请求得到的数据都一样的，我们可以把这对「请求-响应」的数据都缓存在本地，那么下次就直接读取本地的数据，不必在通过网络获取服务器的响应了。这就是HTTP缓存，HTTP 缓存有两种实现方式，分别是强制缓存和协商缓存。它们的区别是强制缓存如果有效，则不需要再向服务器发送请求，直接使用本地缓存，而协商缓存不管是否有效，都需要向服务器发出请求，确认是否使用本地缓存。

强制缓存 ​

强制缓存是指，在浏览器向服务器首次请求数据后，会将数据和缓存标识一起存储在本地缓存中。在下次请求相同资源时，浏览器会首先检查本地缓存中是否存在该资源，以及缓存是否过期。如果缓存有效且未过期，则直接使用本地缓存，不再向服务器发送请求。决定是否使用缓存的主动性在于浏览器这边。

如下图中，返回的是 200 状态码，但在 size 项中标识的是 from disk cache，就是使用了强制缓存。

强制缓存的标识是在HTTP响应头中设置的，主要包括Expires和Cache-Control两种。

  • Expires：这是HTTP/1.0的头部信息，表示资源过期时间，其值为一个绝对时间的GMT格式字符串。但由于其使用的是绝对时间，如果服务器时间和客户端时间存在偏差，就可能导致缓存失效，因此HTTP/1.1开始使用Cache-Control替代。
  • Cache-Control：表示资源在客户端本地缓存后多少秒内有效。该值是一个相对时间。

如果 HTTP 响应头部同时有 Cache-Control 和 Expires 字段的话，Cache-Control 的优先级高于 Expires 。

Cache-control 选项更多一些，设置更加精细，所以建议使用 Cache-Control 来实现强缓存。具体的实现流程如下：

  1. 当浏览器第一次请求访问服务器资源时，服务器会在返回这个资源的同时，在 Response 头部加上 Cache-Control，Cache-Control 中设置了过期时间大小；
  2. 浏览器再次请求访问服务器中的该资源时，会先通过请求资源的时间与 Cache-Control 中设置的过期时间大小，来计算出该资源是否过期，如果没有，则使用该缓存，否则重新请求服务器；
  3. 服务器再次收到请求后，会再次更新 Response 头部的 Cache-Control。

协商缓存 ​

协商缓存，从字面意思上理解，就是需要和服务器进行协商，看看是否可以使用本地缓存。每次请求时，浏览器都会向服务器发送一个请求，询问服务器资源是否有更新，如果有更新，则下载最新的资源；否则，使用本地缓存。

协商缓存的标识也是在HTTP响应头中设置的，主要包括Last-Modified/If-Modified-Since和ETag/If-None-Match两组头部信息。

  • Last-Modified和If-Modified-Since：浏览器在第一次访问资源时，服务器返回资源的同时，在响应头中会加上Last-Modified，表示该资源在服务器上的最后修改时间。浏览器再次访问该资源时，会在请求头中加上If-Modified-Since（值为之前返回的Last-Modified值）。服务器收到请求后，会根据这个时间和资源在服务器上的最后修改时间进行比较，如果一致，则返回304，表示资源未修改，可以使用本地缓存；否则返回200和新的资源内容。
  • ETag和If-None-Match：ETag是服务器根据资源内容生成的一个唯一标识，只要资源内容有变化，ETag就会重新生成。浏览器在收到资源后，也会在本地缓存中保存这个ETag。在下次请求时，浏览器会将ETag值放到If-None-Match请求头中发送给服务器，服务器会将该值与资源当前的ETag进行比较，如果相同，则返回304，表示资源未修改，可以使用本地缓存；否则返回200和新的资源内容。

第一种实现方式是基于时间实现的，第二种实现方式是基于一个唯一标识实现的，相对来说后者可以更加准确地判断文件内容是否被修改，避免由于时间篡改导致的不可靠问题。同时ETag的优先级更高！

为什么 ETag 的优先级更高？

  1. 在没有修改文件内容情况下文件的最后修改时间可能也会改变(比如修改之后，又修改回来)，这会导致客户端认为这文件被改动了，从而重新请求；
  2. 可能有些文件是在秒级以内修改的，If-Modified-Since 能检查到的粒度是秒级的，使用 Etag就能够保证这种需求下客户端在 1 秒内能刷新多次；
  3. 有些服务器不能精确获取文件的最后修改时间。

注意，协商缓存这两个字段都需要配合强制缓存中 Cache-Control 字段来使用，只有在未能命中强制缓存的时候，才能发起带有协商缓存字段的请求。

HTTP 特性 ​

HTTP 常见到版本有 HTTP/1.1，HTTP/2.0，HTTP/3.0，不同版本的 HTTP 特性是不一样的。这里主要以HTTP/1.1为例。

优点 ​

1、简单

HTTP 基本的报文格式就是 header + body，头部header信息也是 key-value简单文本的形式，易于理解，降低了学习和使用的门槛。

2、灵活

HTTP的URL、状态码没有固定死，运行开发者自定义和扩充。同时 HTTP 由于是工作在应用层（ OSI 第七层），则它下层可以随意变化

  • HTTPS 就是在 HTTP 与 TCP 层之间增加了 SSL/TLS 安全传输层；
  • HTTP/1.1 和 HTTP/2.0 传输协议使用的是 TCP 协议，而到了 HTTP/3.0 传输协议改用了 UDP 协议。

3、应用广泛跨平台

缺点 ​

1、无状态

无状态指的是两次不同的HTTP请求之间是没有关联的。其实这是一把双刃剑，一方面无状态可以让服务器不用额外的资源去记录状态的信息；另一方面服务器没有记忆能力，操作一些相关联的请求的时候，会不方便。

对于无状态的问题，解法方案有很多种，其中比较简单的方式用 Cookie 技术。Cookie 通过在请求和响应报文中写入 Cookie 信息来控制客户端的状态。相当于，在客户端第一次请求后，服务器会下发一个装有客户信息的「小贴纸」，后续客户端请求服务器的时候，带上「小贴纸」，服务器就能认得了了，

2、明文传输

一方面，明文意味着在传输过程中的信息，是可方便阅读的，我们抓包很方便；另一方面，HTTP的信息都暴露出去了，没有隐私可言。

3、不安全

  • 通信使用明文（不加密）
  • 不验证通信方的身份
  • 无法证明报文的完整性

性能 ​

1. 长连接

2. 管道网络传输

HTTP/1.1 采用了长连接的方式，可在同一个 TCP 连接里面，客户端可以发起多个请求，只要第一个请求发出去了，不必等其回来，就可以发第二个请求出去，可以减少整体的响应时间。

但是服务器必须按照接收请求的顺序发送对这些管道化请求的响应。如果服务端在处理 A 请求时耗时比较长，那么后续的请求的处理都会被阻塞住，这称为「队头堵塞」。所以，HTTP/1.1 管道解决了请求的队头阻塞，但是没有解决响应的队头阻塞。

TIP

实际上 HTTP/1.1 管道化技术不是默认开启，而且浏览器基本都没有支持

3、队头阻塞

HTTP和HTTPS ​

HTTP和HTTPS的区别 ​

  1. HTTP 明文传输，有安全问题；HTTPS 在 TCP 和 HTTP 网络层之间加入了 SSL/TLS 安全协议，使得报文能够加密传输。
  2. HTTP 连接建立简单， TCP 三次握手之后便可进行 HTTP 的报文传输； HTTPS 在 TCP 三次握手之后，还需进行 SSL/TLS 的握手过程，才可进入加密报文传输。
  3. 端口不同，分别是80端口和443端口
  4. HTTP不需要向CA申请证书；HTTPS需要申请证书

HTTPS 解决了 HTTP 的哪些问题？ ​

  1. 窃听风险。（因为传输是明文的）
  2. 篡改风险。
  3. 冒充风险

HTTPS 在 HTTP 与 TCP 层之间加入了 SSL/TLS 协议，可以很好的解决了上述的风险：

  1. 信息加密：交互信息无法被窃取。
  2. 校验机制：无法篡改通信内容，篡改了就不能正常显示。
  3. 身份证书：证明网站是真实的。

HTTPS工作流程 ​

1、混合加密

这是一种结合了对称加密和非对称加密的加密方式。分为两个阶段：

  1. 密钥交换阶段：在这个阶段，HTTPS使用非对称加密来安全地交换对称密钥。
    • 客户端向服务器发送一个连接请求，请求建立HTTPS连接。
    • 服务器接收到请求后，将自己的公钥发送给客户端。
    • 客户端使用服务器的公钥对一个随机生成的对称密钥进行加密，并将加密后的对称密钥发送给服务器。
    • 服务器使用其私钥对接收到的加密对称密钥进行解密，获取到对称密钥。
    • 至此，客户端、服务器都有了一样的对称密钥。
  2. 数据通信阶段：在密钥交换完成后，HTTPS使用对称密钥进行实际的数据通信。
    • 客户端和服务器使用之前交换的对称密钥进行加密和解密操作，确保通信内容的安全性。
    • 对称密钥只在当前会话中使用，保证了数据的安全性。

2. 摘要算法 + 数字签名

  1. 摘要算法（Message Digest）：

    • 摘要算法是一种哈希函数，它将任意长度的数据转换为固定长度的哈希值（也称为消息摘要或指纹）。这个哈希值是唯一的，且无法通过哈希值推导出内容。

    • 在HTTPS中，发送方会将待发送的数据通过摘要算法进行处理，生成一个固定长度的哈希值。

    • 这个哈希值随数据一起发送给接收方。

    • 接收方收到数据后，同样使用相同的摘要算法对数据进行处理，生成一个新的哈希值。

    • 接收方比较收到的哈希值和自己生成的哈希值，如果两者相同，则说明数据在传输过程中没有被篡改，保证了数据的完整性。

  1. 数字签名（Digital Signature）：

    • 数字签名是一种用于验证消息来源和完整性的技术。

    • 首先，发送方使用摘要算法对待发送的数据生成一个哈希值。

    • 然后，发送方使用自己的私钥对这个哈希值进行加密，生成数字签名。

    • 数字签名和原始数据一起发送给接收方。

    • 接收方收到数据后，使用发送方的公钥对数字签名进行解密，得到哈希值。

    • 同时，接收方也对收到的数据使用相同的摘要算法生成一个新的哈希值。

    • 接收方比较这两个哈希值，如果它们相同，则说明数据是由发送方发送的，并且数据在传输过程中没有被篡改，保证了数据的完整性和认证。

私钥加密，公钥解密。通常是为了确认发送者的身份。

3、数字证书

其实服务器已经可以确定发送者的身份了，但是我们如果能修改掉服务器的"公钥"，再用自己的私钥去发送数据，那么就又可以伪造了。

既然伪造公私钥那么随意，所以服务器把自己的公钥注册到CA，CA用他们自己的私钥对服务器的公钥做了个数字签名，然后把服务器的「个人信息 + 公钥 + 数字签名」打包成一个数字证书，也就是说这个数字证书包含服务器的公钥。

客户端拿到了数字证书后，首先会去CA验证这个数字证书是否合法。

HTTPS如何建立连接 ​

SSL/TLS 协议基本流程：

  • 客户端向服务器索要并验证服务器的公钥。
  • 双方协商生产「会话秘钥」。
  • 双方采用「会话秘钥」进行加密通信。

前两步也就是 SSL/TLS 的建立过程，也就是 TLS 握手阶段。

TLS（Transport Layer Security）协议是用于在互联网上安全地传输数据的协议，它是SSL（Secure Sockets Layer）协议的后继者。TLS协议建立连接的过程，通常称为TLS握手，包括以下步骤：

  1. 客户端发起连接：
    • 客户端发送一个带有TLS版本号、随机数Client Random和支持的加密算法的ClientHello消息到服务器。
  2. 服务器响应：
    • 服务器发送一个ServerHello消息，其中包括选定的TLS版本、随机数Server Random、加密算法、服务器证书（包含服务器的公钥）以及其他必要的信息。
  3. 客户端验证证书：
    • 客户端验证服务器证书的有效性，包括证书链的信任、证书有效期、证书是否被撤销等。
    • 如果证书验证失败，客户端可以终止连接。
  4. 生成会话密钥：
    • 客户端生成一个随机的会话密钥（对称密钥），并使用服务器的公钥将其加密，发送给服务器。
    • 这个会话密钥将用于加密和解密后续通信中的数据。
  5. 密钥交换：
    • 服务器使用自己的私钥解密会话密钥。
    • 此时，客户端和服务器都拥有了相同的会话密钥。
  6. 客户端就绪：
    • 客户端发送一个ChangeCipherSpec消息，表示随后的消息将使用协商好的加密算法和会话密钥进行加密。
    • 客户端还会发送一个包含加密的握手消息哈希（Finished message）的消息，以供服务器验证。
  7. 服务器就绪：
    • 服务器同样发送一个ChangeCipherSpec消息，表示随后的消息也将使用协商好的加密算法和会话密钥进行加密。
    • 服务器发送一个包含加密的握手消息哈希（Finished message）的消息，以供客户端验证。
  8. 握手完成：
    • 客户端验证服务器的Finished message，确认握手过程没有被篡改。
    • 一旦验证通过，客户端和服务器都认为握手成功，并且可以使用协商好的加密算法和会话密钥开始安全通信。
  9. 安全通信：
    • 握手完成后，双方可以使用会话密钥和安全协商的加密算法来加密和解密数据，确保数据的机密性和完整性。

TLS握手过程确保了客户端和服务器之间的安全连接，防止了数据在传输过程中被窃取或篡改。这个过程还支持证书的验证，确保客户端正在与正确的服务器进行通信。

总结 ​

HTTPS（Hypertext Transfer Protocol Secure）是HTTP的安全版本，它在传输数据时使用SSL（Secure Sockets Layer）或TLS（Transport Layer Security）协议来加密数据。以下是HTTPS的整个工作流程：

  1. 客户端发起连接：
    • 客户端（通常是浏览器）通过输入HTTPS网址或点击一个HTTPS链接来发起一个到服务器的连接请求。
  2. 服务器响应并提供证书：
    • 服务器响应客户端的连接请求，并提供它的数字证书。这个证书通常由第三方可信的证书颁发机构（CA）签发，包含服务器的公钥和一些元数据。
  3. 客户端验证证书：
    • 客户端使用内置的CA公钥来验证服务器证书的有效性。这包括检查证书是否由受信任的CA签发、是否过期、是否被撤销，以及证书中的域名是否与请求的域名匹配。
  4. 生成临时密钥和密钥交换：
    • 客户端生成一个随机的密钥，称为“会话密钥”或“对称密钥”。客户端使用服务器的公钥对这个会话密钥进行加密，然后发送给服务器。
    • 服务器使用自己的私钥解密这个会话密钥。此时，客户端和服务器都拥有相同的会话密钥，它可以用来加密和解密接下来传输的数据。
  5. 安全协商和加密通信：
    • 双方使用这个会话密钥和协商好的加密算法，对传输的数据进行加密和解密。
    • 此外，双方还会协商一个安全的哈希函数，用于生成数据的哈希值（消息摘要），以确保数据的完整性。
  6. 数据传输：
    • 一旦安全连接建立，客户端和服务器就可以开始安全地传输数据。所有通过这个连接传输的数据都会被加密，确保了数据的机密性和完整性。
  7. 连接结束：
    • 当数据传输完成后，连接可以被关闭。在连接的生命周期内，所有的数据都是通过加密通道传输的，防止了窃听和中间人攻击。

总结来说，HTTPS的工作流程包括客户端和服务器之间的安全握手，证书的验证，临时密钥的生成和交换，以及使用加密算法和哈希函数来保护数据的安全传输。通过这些机制，HTTPS确保了数据在互联网上传输时的隐私和完整性。
