Java并发编程 ​

基础 ​

​

并发和并行的区别 ​

  • 并发（Concurrent）：指在同一个时间段内，两个或多个程序（或任务、事件）执行，它们在时间上有重叠，即宏观上是同时进行的，但在微观上仍然是顺序执行的。在单处理机系统中，并发程序通过时间片轮转等方式交替执行，任一时刻只有一个程序在处理机上运行。
  • 并行（Parallel）：指两个或者多个程序（或任务、事件）在同一时刻同时发生，它们互不干扰地独立执行。在多处理机系统中，每个处理机可以独立处理一个任务，实现真正的并行执行。

进程和线程 ​

进程（Process）：

  1. 定义：进程是计算机中的程序关于某数据集合上的一次运行活动，是系统进行资源分配的基本单位。它是一个动态的概念，包括程序的代码以及当前的活动。
  2. 特点
    • 动态性：进程是程序的一次执行过程，动态产生并消亡。
    • 并发性：进程可以与其他进程一起并发执行。
    • 独立性：进程是独立运行的基本单位，系统为其分配独立的资源。
    • 异步性：进程按各自独立的、不可预知的速度向前推进。
  3. 结构：进程由程序、数据和进程控制块（PCB）三部分组成。

线程（Thread）：

  1. 定义：线程是操作系统能够进行运算调度的最小单位，包含在进程之中，是进程中的实际运作单位，多个线程可以并发执行不同的任务。
  2. 特点
    • 轻型实体：线程基本不拥有系统资源，只拥有一点必不可少的资源（如程序计数器、一组寄存器和栈）。
    • 独立调度和分派：线程是独立调度和分派的基本单位，可以由操作系统内核调度，也可以由用户进程自行调度。
    • 共享进程资源：同一进程中的多条线程共享该进程的全部系统资源，如虚拟地址空间、文件描述符等，但拥有各自的调用栈、寄存器环境和线程本地存储。
  3. 优势：在多核或多CPU环境中，多线程程序设计可以提高程序的执行吞吐率；在单核环境中，也可以提高程序的执行效率，特别是将I/O密集型和CPU密集型任务分开处理时。

总的来说，进程是操作系统分配资源的基本单位，而线程是操作系统调度的基本单位。线程共享进程的资源，使得线程的创建、切换和销毁的开销相对较小，从而提高了系统的并发性和执行效率。

协程 ​

协程（coroutine）是一种程序组件，它可以被理解为一种用户态的轻量级线程。它允许程序在执行过程中进行挂起和恢复操作。这种挂起和恢复的操作是由程序自身控制的。

  • 轻量级：协程的创建、切换和销毁的开销远小于线程，因为它是在用户态进行切换的，无需内核的参与。
  • 协作式调度：协程的调度是由程序自身控制的，当协程遇到阻塞操作（如IO操作）时，它可以主动挂起，让出执行权给其他协程。
  • 并发性：虽然协程在同一时间只能执行一个，但它们通过挂起和恢复的操作可以在逻辑上实现并发执行的效果。这种并发性对于处理IO密集型任务尤为有效。

线程的共享内存 ​

线程之间想要进行通信，可以通过消息传递和共享内存两种方法来完成。那 Java 采用的是共享内存的并发模型。

线程 A 与线程 B 之间如要通信的话，必须要经历下面 2 个步骤：

  • 线程 A 把本地内存 A 中的共享变量副本刷新到主内存中。
  • 线程 B 到主内存中读取线程 A 刷新过的共享变量，再同步到自己的共享变量副本中。

java
class MyThread extends Thread {
    public void run() {
        System.out.println("线程运行中...");

}

public class ThreadExample {
    public static void main(String[] args) {
        MyThread myThread = new MyThread();
        myThread.start();

  • 垃圾回收线程，它是一个后台线程，负责回收不再使用的对象。

线程的调度方法 ​

等待与通知是一对，在等待方法中，包括：

  1. wait()：当一个线程 A 调用一个共享变量的 wait() 方法时，线程 A 会被阻塞挂起，直到发生下面几种情况才会返回 ：

    • 线程 B 调用了共享对象 notify()或者 notifyAll() 方法；

    • 其他线程调用了线程 A 的 interrupt() 方法，线程 A 抛出 InterruptedException 异常返回。

  2. wait(long timeout) ：这个方法相比 wait() 方法多了一个超时参数，它的不同之处在于，如果线程 A 调用共享对象的 wait(long timeout)方法后，没有在指定的 timeout 时间内被其它线程唤醒，那么这个方法还是会因为超时而返回。

  3. wait(long timeout, int nanos)，其内部调用的是 wait(long timout) 方法。

在通知方法中，包括：

  1. notify()：一个线程 A 调用共享对象的 notify() 方法后，会唤醒一个在这个共享变量上调用 wait 系列方法后被挂起的线程。不过一个共享变量上可能会有多个线程在等待，具体唤醒哪个等待的线程是随机的。
  2. notifyAll()：不同于在共享变量上调用 notify() 方法会唤醒被阻塞到该共享变量上的一个线程，notifyAll 方法会唤醒所有在该共享变量上调用 wait 系列方法而被挂起的线程。

Thread 类还提供了一个 join() 方法，意思是如果一个线程 A 执行了 thread.join()，当前线程 A 会等待 thread 线程终止之后才从 thread.join() 返回。

在让出优先权方法中：

  1. yield()：Thread 类中的静态方法，当一个线程调用 yield 方法时，实际是在暗示线程调度器，当前线程请求让出自己的 CPU，但是线程调度器可能会“装看不见”忽略这个暗示。

在线程中断中，主要有3个相关的方法：

  1. interrupt() 方法：
    • 这是一个实例方法，用于向线程发送中断信号。它设置线程的中断状态，并且线程应该通过检查这个中断状态来响应中断。
    • 调用线程的 interrupt() 方法不会自动停止线程，它只是设置线程的中断状态。线程需要自己检查这个状态，并决定如何响应中断。
    • 线程可以通过 Thread.currentThread().isInterrupted() 来检查自己的中断状态。
  2. interrupted() 方法：
    • 这是一个静态方法，用于检查当前线程的中断状态，并在检查后清除这个状态。
    • 如果当前线程已被中断，则该方法返回 true，否则返回 false。
    • 由于 interrupted() 方法会清除中断状态，所以通常只在需要清除状态并立即处理中断的情况下使用。
  3. isInterrupted() 方法：
    • 这是一个实例方法，用于检查线程的中断状态，但不会清除这个状态。
    • 如果线程已被中断，则该方法返回 true，否则返回 false。
    • 与 interrupted() 不同，isInterrupted() 不会改变线程的中断状态，所以可以在多个地方多次调用以检查状态。

线程池 ​

什么是线程池 ​

线程池是一种多线程处理形式，旨在优化线程的使用和管理。

线程池的优势如下：

  • 降低系统资源消耗：通过重用已存在的线程，减少线程创建和销毁造成的消耗。
  • 提高系统响应速度：任务到达时，无需等待新线程的创建便能立即执行。
  • 方便线程并发数的管控：避免无限制地创建线程，从而防止系统资源过度消耗和潜在的稳定性问题。
  • 提供更强大的功能：例如定时、定期执行任务等。

线程池的核心参数 ​

  1. 核心线程数（Core Pool Size）：这是线程池中始终维护的线程数量，即使这些线程处于空闲状态也不会被销毁，除非设置了allowCoreThreadTimeOut。
  2. 最大线程数（Maximum Pool Size）：线程池允许创建的最大线程数。当工作队列满了之后，如果还有任务提交，线程池会创建新的线程，直到达到这个数。
  3. 工作队列（Work Queue）：用于存储待执行任务的队列。当核心线程都在使用中，新来的任务会存储在这个队列中等待处理。
  4. 线程存活时间（Keep Alive Time）：当线程数超过核心线程数时，这是多余空闲线程在终止前等待新任务的最大时间。
  5. TimeUnit（时间单位）：与keepAliveTime一起使用，用于指定时间的单位，如秒、毫秒、微秒等。
  6. 线程工厂（Thread Factory）：用于创建新线程的工厂，可以用来设置线程的名称、线程组、优先级等。
  7. 拒绝策略（Rejected Execution Handler）：当工作队列满了并且线程数达到最大线程数时，对新来的任务采取的处理策略。常见的拒绝策略有：
    • 直接抛出异常（AbortPolicy）
    • 直接由调用者本身调用run方法，并且是阻塞执行的（CallerRunsPolicy）
    • 直接丢弃任务，不做任何处理（DiscardPolicy）
    • 丢弃队列中最旧的任务并执行当前任务（DiscardOldestPolicy）

使用示例如下

java
import java.util.concurrent.*;  
  
public class ThreadPoolExecutorExample {  
    public static void main(String[] args) {  
        // 定义线程池参数  
        int corePoolSize = 5; // 核心线程数  
        int maximumPoolSize = 10; // 最大线程数  
        long keepAliveTime = 60L; // 线程空闲时间，单位：秒  
        TimeUnit unit = TimeUnit.SECONDS; // 时间单位  
        BlockingQueue<Runnable> workQueue = new ArrayBlockingQueue<>(100); // 工作队列  
        ThreadFactory threadFactory = Executors.defaultThreadFactory(); // 线程工厂，使用默认工厂  
        RejectedExecutionHandler handler = new ThreadPoolExecutor.AbortPolicy(); // 拒绝策略，使用默认策略  
  
        // 创建 ThreadPoolExecutor 实例  
        ThreadPoolExecutor executor = new ThreadPoolExecutor(  
                corePoolSize,  
                maximumPoolSize,  
                keepAliveTime,  
                unit,  
                workQueue,  
                threadFactory,  
                handler  
        );  
  
        // 提交任务到线程池执行  
        for (int i = 0; i < 15; i++) {  
            Runnable worker = new WorkerThread("" + i);  
            executor.execute(worker); // 执行任务  
        }  
  
        // 关闭线程池（通常不会立即关闭，因为可能存在未完成的任务）  
        executor.shutdown();  
        while (!executor.isTerminated()) {  
        }  
  
        System.out.println("所有任务已完成");  
    }  
  
    static class WorkerThread implements Runnable {  
        private String command;  
  
        public WorkerThread(String s) {  
            this.command = s;  
        }  
  
        @Override  
        public void run() {  
            System.out.println(Thread.currentThread().getName() + " 开始执行命令 : " + command);  
            processCommand();  
            System.out.println(Thread.currentThread().getName() + " 结束执行命令");  
        }  
  
        private void processCommand() {  
            try {  
                Thread.sleep(2000);  
            } catch (InterruptedException e) {  
                e.printStackTrace();  
            }  
        }  
  
        @Override  
        public String toString() {  
            return this.command;  
        }  
    }  
}

线程池的submit方法和execute方法 ​

submit 方法和 execute 方法是 java.util.concurrent.ExecutorService 接口中的两个重要方法，用于执行异步任务。它们的主要区别在于：

  1. 返回值：
    • submit 方法会返回一个 Future 对象，通过这个对象，我们可以了解任务执行的情况（是否完成、是否被取消等），并且可以通过 Future 的 get 方法获取任务的返回结果，如果任务执行过程中发生异常，get 方法会抛出 ExecutionException。
    • execute 方法没有返回值，它只是执行任务，但是无法获取任务执行的结果或状态。
  2. 异常处理：
    • 使用 submit 方法提交的任务，如果任务在执行过程中出现异常，异常会被包装成 ExecutionException 通过 Future 对象抛出，可以在调用 Future.get() 方法时捕获并进行处理。
    • 使用 execute 方法提交的任务，如果任务在执行过程中出现异常，异常只能在任务内部进行处理，或者通过为线程设置 UncaughtExceptionHandler 来处理未捕获的异常。
  3. 使用场景：
    • 当你需要获取任务执行的结果时，应该使用 submit 方法。
    • 当你只关心任务被执行，而不关心任务执行的结果时，可以使用 execute 方法。

示例代码：

使用 submit 方法：

Java
ExecutorService executorService = Executors.newFixedThreadPool(10);
Future<String> future = executorService.submit(() -> {
    // 执行任务
    return "Result";
});

try {
    String result = future.get(); // 获取任务执行结果
} catch (InterruptedException | ExecutionException e) {
    e.printStackTrace();
}

线程池的关闭方法 ​

Java中的线程池可以通过调用ExecutorService接口提供的shutdown和shutdownNow方法来关闭。这两个方法的行为有所不同：

  1. shutdown方法：
    • 这个方法会平滑地关闭线程池，它允许已经提交到线程池的任务执行完成，但不接受新任务。
    • 一旦所有任务都执行完毕，线程池中的线程会一个接一个地关闭。
    • 如果在关闭过程中有新的任务提交，这些任务会被拒绝，并且调用者可以通过isShutdown方法检查线程池是否已经关闭。
  2. shutdownNow方法：
    • 这个方法会尝试立即关闭线程池，它会尝试中断所有正在执行的线程，并且返回尚未开始执行的任务列表。
    • 这个方法返回的未执行任务列表可以被用来进行一些后续的处理，比如记录日志或者保存起来稍后重试。
    • isShutdown和isTerminated方法可以用来判断线程池是否已经进入关闭状态以及是否已经完全终止。

这里补充说明一下isShutdown 和 isTerminated 方法。

isShutdown 和 isTerminated 是 ExecutorService 接口中的两个方法，用于检查线程池的状态。

  1. isShutdown 方法：
    • 这个方法用于判断线程池是否已经开始了关闭过程。如果调用了 shutdown 或 shutdownNow 方法，那么 isShutdown 方法将返回 true。
    • 即使线程池还在执行关闭过程中的任务，isShutdown 方法也会返回 true，因为它表示的是线程池已经接收到关闭的请求。
    • 一旦线程池完全关闭，这个方法仍然会返回 true。
  2. isTerminated 方法：
    • 这个方法用于判断线程池是否已经完全终止。线程池完全终止的条件是所有的任务都已经执行完毕，并且所有的线程都已经结束。
    • 当线程池处于终止状态时，isTerminated 方法返回 true。
    • 如果线程池还没有完全关闭，即使 shutdown 或 shutdownNow 方法已经被调用，isTerminated 方法也会返回 false。

通常，如果希望已经提交的任务能够执行完成，你应该使用shutdown方法。如果你需要立即停止所有任务并且不再关心它们的执行结果，那么可以使用shutdownNow方法。

java
ExecutorService executorService = Executors.newFixedThreadPool(10);

// ... 提交任务到线程池 ...

// 关闭线程池
executorService.shutdown(); // 平滑关闭

// 或者立即关闭
List<Runnable> unfinishedTasks = executorService.shutdownNow(); // 立即关闭

// 等待线程池完全关闭
try {
    if (!executorService.awaitTermination(800, TimeUnit.MILLISECONDS)) {
        executorService.shutdownNow(); // 如果关闭超时，尝试立即关闭
    }
} catch (InterruptedException e) {
    executorService.shutdownNow(); // 如果等待过程中被中断，尝试立即关闭
}

// 处理未完成的任务
for (Runnable task : unfinishedTasks) {
    // ... 执行一些操作，比如记录日志或者保存任务 ...
}

线程池的工作队列 ​

在Java中，常用的阻塞队列主要有以下几种：

  1. ArrayBlockingQueue：一个由数组结构组成的有界阻塞队列。
  2. LinkedBlockingQueue：一个由链表结构组成的有界阻塞队列，但容量可以选择进行设置，不设置的话，就是 Integer.MAX_VALUE，容量非常大，因此也常常被当做无界队列使用。
  3. PriorityBlockingQueue：一个支持优先级排序的无界阻塞队列。
  4. DelayQueue：一个支持延时获取元素的无界阻塞队列，即可以指定多久才能从队列中获取当前元素。
  5. SynchronousQueue：一个不存储元素的阻塞队列，即直接提交策略。每个插入操作必须等到另一个线程调用移除操作，否则插入操作一直处于阻塞状态。

当提交任务的速度大于处理任务的速度时，每次提交一个任务，就必然会创建一个线程。极端情况下会创建过多的线程，耗尽 CPU 和内存资源。由于空闲 60 秒的线程会被终止，长时间保持空闲的 CachedThreadPool 不会占用任何资源。

工作流程：

  • 提交任务
  • 因为没有核心线程，所以任务直接加到 SynchronousQueue 队列。
  • 判断是否有空闲线程，如果有，就去取出任务执行。
  • 如果没有空闲线程，就新建一个线程执行。
  • 执行完任务的线程，还可以存活 60 秒，如果在这期间，接到任务，可以继续活下去；否则，被销毁。

SynchronousQueue 是一个没有容量的阻塞队列，常用于线程之间的直接传递。每个插入操作必须等待一个相应的删除操作，反之亦然。它的使用示例如下

Details
java
public class SynchronousQueueExample {

    public static void main(String[] args) {
        // 创建一个SynchronousQueue实例
        SynchronousQueue<Integer> queue = new SynchronousQueue<>();

        // 创建一个生产者线程
        Thread producerThread = new Thread(() -> {
            try {
                for (int i = 0; i < 5; i++) {
                    System.out.println("生产者线程放入: " + i);
                    queue.put(i); // 将元素放入队列，如果没有消费者取走会阻塞
                    Thread.sleep(1000); // 模拟生产过程
                }
            } catch (InterruptedException e) {
                e.printStackTrace();
            }
        });

        // 创建一个消费者线程
        Thread consumerThread = new Thread(() -> {
            try {
                for (int i = 0; i < 5; i++) {
                    Integer value = queue.take(); // 从队列中取出元素，如果没有会阻塞等待
                    System.out.println("消费者线程取出: " + value);
                }
            } catch (InterruptedException e) {
                e.printStackTrace();
            }
        });

        // 启动消费者线程，如果取不出来，会被阻塞
        consumerThread.start();

        try {
            Thread.sleep(3000);
        } catch (InterruptedException e) {
            throw new RuntimeException(e);
        }
        producerThread.start();
    }
}

  1. UncaughtExceptionHandler： 对于未被捕获的异常，可以通过设置Thread.UncaughtExceptionHandler来处理。在Java中，UncaughtExceptionHandler 是一个接口，它定义了一个方法 uncaughtException(Thread t, Throwable e)，用于处理未被捕获的异常。这个方法会在线程因未捕获的异常而突然终止时被调用。

你可以通过 Thread.setUncaughtExceptionHandler 方法来设置线程的未捕获异常处理器。以下是一个简单的例子，展示了如何使用 UncaughtExceptionHandler：

java
Thread.UncaughtExceptionHandler handler = new Thread.UncaughtExceptionHandler() {  
    @Override  
    public void uncaughtException(Thread t, Throwable e) {  
        // 在这里处理异常，例如记录日志、重新尝试任务、通知管理员等  
        System.err.println("线程 " + t.getName() + " 出现异常: " + e.getMessage());  
        e.printStackTrace();  
        // 可以根据实际需求进行更复杂的异常处理  
    }  
};  

ThreadPoolExecutor executor = (ThreadPoolExecutor) Executors.newCachedThreadPool();  
// 为线程池中的每个线程设置 UncaughtExceptionHandler  
for (Thread t : executor.getPoolSize()) {  
    t.setUncaughtExceptionHandler(handler);  
}

线程池的状态 ​

线程池主要有以下五种状态：

  1. RUNNING
    • 表示线程池处于正常运行状态。
    • 线程池被创建后一直处于此状态。
    • 在此状态下，线程池能够接受新任务，以及对已添加的任务进行处理。
  2. SHUTDOWN
    • 表示线程池正在关闭状态。
    • 当调用shutdown()方法后，线程池会进入此状态。
    • 在此状态下，线程池不会接受新任务，但会继续处理已经提交的任务。
  3. STOP
    • 表示线程池处于停止状态。
    • 当调用shutdownNow()方法后，线程池会进入此状态。
    • 在此状态下，线程池不会接受新任务，并且会尝试停止所有正在执行的任务。
  4. TIDYING
    • 表示线程池正在整理状态。
    • 当所有任务都已经终止，工作线程数量为零时，线程池会进入此状态。
    • 在此状态下，线程池会执行一些清理工作，并准备终止。
  5. TERMINATED
    • 表示线程池已经终止状态。
    • 在整理完毕后，线程池会进入此状态。
    • 在此状态下，线程池已经完全停止，不再接受新任务，也不会处理任何任务。
